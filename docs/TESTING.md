# AiRemedy(AI신약) MCP — 테스트 계획 및 결과

MCP 서버(툴 14개)를 **카카오 PlayMCP가 실제로 부르는 방식 그대로** 검증한다.

이 문서의 원칙은 하나다: **REST로 통과한 테스트는 아무것도 증명하지 못한다.** 실제로 프로덕션에 도달한 결함은 전부 REST에서 보이지 않았다 — 응답 크기, 툴 설명이 모델을 이끄는 방식, `isError` 플래그, 10초 타임아웃, 동시 요청. 그래서 모든 테스트는 `/mcp` Streamable HTTP 위에서, `tools/call`로 수행한다.

## 하네스

| 스크립트 | 질문 |

`kakao_sim.py`가 핵심이다. **우리 시스템 프롬프트를 0줄 준다** — 카카오에는 플레이그라운드의 정교한 프롬프트가 없기 때문이다. 프롬프트 덕분에만 성립하는 동작은 프로덕션에서 성립하지 않는다.

```bash
python scripts/mcp_bench.py                  # 지연 + 응답 크기
python scripts/kakao_sim.py --scenario vaccine
python scripts/mcp_selection.py              # 툴 선택 정확도
python scripts/mcp_stress.py fuzz -n 20
python scripts/mcp_stress.py chain | concurrent -n 15 | cache | params | isolation | deterministic
HTTP_CHAOS_DELAY_MS=4000 uvicorn main:app ...   # 카오스: 지연 주입
HTTP_CHAOS_FAIL_RATE=0.6 uvicorn main:app ...   # 카오스: 강제 실패
```

## Test 1 — 툴 선택 정확도 & Test 17 — 할루시네이션

정상 질의 11건 + 적대적 입력 9건(10,000자, gibberish, SQL 인젝션, 이모지, 프롬프트 인젝션, 의료 처방 요구, 없는 툴 호출 요구).

**적대적 입력의 정답은 "아무 툴도 부르지 않는 것"이다.** `asdfasdf`에 툴을 호출하는 건 어깨를 으쓱하는 것보다 나쁘다.

**결과: 20/20 (100%), 능력 날조 0건.**

> ⚠️ **할루시네이션 0에 대한 정직한 단서**: OpenAI function calling에서는 모델이 목록에 없는 툴 이름을 **물리적으로 낼 수 없다.** 0은 구조적으로 보장된 것이지 획득한 것이 아니며, 이를 "통과한 테스트"로 보고하는 건 연극이다. 실제로 검증할 가치가 있는 건 **툴이 없을 때 산문으로 능력을 날조하는가**이고(`protein_magic_tool 분석을 실행했습니다`), 그것을 측정했다 — 0건.

**이 테스트가 잡은 실제 결함 (최초 70% → 100%):**
- "폐암 유전자는 어떤 병들과 관련이 있어?" 류 3건에서 모델이 **툴을 안 부르고 기억으로 답했다.** 플레이그라운드는 `tool_choice="required"`로 막지만 **카카오엔 그 안전장치가 없다.** 원인은 `get_target_*` 툴 설명에 유전자→UniProt 매핑이 없어서 모델이 "인자를 못 만드니 그냥 답하자"로 간 것. 설명에 매핑과 "절대 기억으로 답하지 말 것"을 명시해 해결.
- "아스피린이 신약 후보로 적합한지" → `search_similar_compounds`(다른 분자 찾기)를 골랐다. `predict_admet_profile` 설명에 용도("이 약이 안전한가?")를 명시해 해결.

## Test 2 — 랜덤 퍼징 (20회)

빈 문자열, 공백, 10,000자, SQL 인젝션, XSS, 경로 순회(`../../etc/passwd`), 널바이트, 이모지, 템플릿 인젝션, 유전자명/질병명을 accession 자리에 — 50% 확률로 주입.

**결과: 프로세스 크래시 0, 10초 초과 0, 최장 5.1초.** 유효 입력 7건은 정상 응답, 악성 입력 13건은 전부 `isError`.

기준은 "전부 성공"이 **아니다**. 나쁜 입력은 **실패해야** 한다. 문제는 MCP 규약대로(`isError` + 모델이 행동할 수 있는 메시지) 실패하느냐, 아니면 매달리거나 크래시하거나 **아무것도 없는데 자신 있게 답하느냐**다. SQL 인젝션 문자열에 "0건 검색됨"이라고 답한 툴은 통과한 게 아니라 **공손하게 거짓말한 것**이다.

**이 테스트가 잡은 실제 결함 (MCP 규약 위반 3건):**
- `generate_decision_report('EGFR')`, `run_sar_optimization('코로나 신약')` → 존재하지 않는 job인데 `isError=false`로 **초록색 성공** 표시. 카카오에서 실패가 성공으로 보이던 원인과 같은 부류.
- `search_clinical_trials('\n\t')` → 빈 질의를 "0건"으로 응답.

## Test 3 — 콜드 캐시 (1→2→3, 5회)

매 회차마다 **프로세스를 새로 띄워** 인메모리 캐시를 전부 비운다: ChEMBL 타겟 ID 캐시, PubChem CID 캐시, HTTP 연결 풀(TLS 핸드셰이크 재수행), 잡 스토어.

> Redis/브라우저 캐시는 **해당 없음**: 이 서버는 Redis를 쓰지 않고 캐시는 전부 프로세스 메모리 안에 있다. 프로세스 재시작이 곧 완전한 flush다. 없는 컴포넌트를 지웠다고 보고하는 건 거짓 보고다.

**최종 결과 (5회 전부, 방해 없는 클린 실행):**

| 회차 | 전체 시간 | ① 백신 설계 | ② 논문·임상 | ③ 리포트 | 타임아웃 |
|---|---|---|---|---|---|
| 1 | 71s | ✅ 5회 폴링 | ✅ | ✅ 3,966자 | 0 |
| 2 | 77s | ✅ 4회 폴링 | ✅ | ✅ 3,977자 | 0 |
| 3 | 69s | ✅ 4회 폴링 | ✅ | ✅ | 0 |
| 4 | 86s | ✅ 5회 폴링 | ✅ | ✅ | 0 |
| 5 | 70s | ✅ 4회 폴링 | ✅ | ✅ 3,985자 | 0 |

**5/5 완주, 10초 타임아웃 0건.** 콜드 스타트에서도 백신 설계(①)는 4~5회 폴링(~10초)으로 한 턴 안에 끝난다.

**이 테스트가 잡은 실제 결함 2건:**
- `search_clinical_trials`가 **10초를 넘겨 죽었다.** 툴의 8.5초 예산이 LLM 요약에만 걸려 있고, ClinicalTrials.gov 조회는 **별도의 8초 예산**을 썼다. 번역(1초) + 조회(8초) + 요약이면 10초를 넘긴다. 조회 예산을 툴 예산 안으로(5초) 넣어 해결.
- 5회 중 2회에서 **리포트 응답이 잘렸다**(4,012·4,015자 → 4,000자 컷). 잘려나간 꼬리는 리포트 파일의 **로컬 절대경로**(`C:\Users\...\reports\….html`)였다 — 카카오 사용자는 접근할 수도 없는 경로이고, 모델은 그걸 성실하게 "아무 데도 가지 않는 링크"로 렌더링했다. **쓸모없으면서 쓸모있는 것까지 밀어내던 필드**라 삭제했다. 리포트 응답 4,015자 → **3,575자**.

> 캐시가 비어 있어도 스탬피드는 없다. 잡은 `job_id`로 직렬화되고, 외부 API 호출은 `http_budget`의 풀 하나(keepalive 20)를 공유하며, 스크리닝 동시성은 세마포어로 제한된다.

## Test 4 — 웜 캐시

| 툴 | 콜드 | 웜 | 개선 |
|---|---|---|---|
| `search_known_inhibitors` | 7.47s | 4.44s | 1.7배 |
| `search_similar_compounds` | 2.11s | 0.39s | **5.4배** |

TTL은 **의도적으로 없다.** 캐시하는 것은 *영구 식별자*뿐이다 — UniProt→ChEMBL 타겟 ID(CHEMBL203은 영원히 EGFR), SMILES→유사 CID 집합(PubChem 인덱스의 성질). 이들은 만료될 이유가 없다. 측정값(친화도·IC50·논문)은 **캐시하지 않는다.**

## Test 5 — 파라미터 검증

필수 누락, 타입 오류, 음수, 과대값, 유전자명↔accession 혼동, 약 이름↔SMILES 혼동, UniProt↔job_id 혼동, 미지의 파라미터 — **9/9 전부 거부.**

**이 테스트가 잡은 실제 결함:** `max_results=-5`가 그대로 통과해 PubMed가 아무것도 안 돌려주고, 툴은 이를 **"0건"**으로 보고했다. 모델은 이걸 **"관련 연구가 존재하지 않는다"는 과학적 주장**으로 읽는다. 잘못된 인자가 **없는 음성 소견을 제조**할 수 있어서는 안 된다.

## Test 8 — 툴 체이닝

`search_known_inhibitors(EGFR)` → 그 SMILES → `predict_admet_profile` + `search_similar_compounds`
`predict_neoantigen_candidates` → job_id → `status` → `generate_vaccine_report`
**모달리티 교차**: 백신 job_id를 저분자 리포트 툴에 → 거부되어야 함

**이 테스트가 잡은 실제 결함 2건:**
- `search_similar_compounds`가 **실제 ChEMBL SMILES를 거부**했다. 이성질체 SMILES에는 `/`와 `\`(이중결합 입체화학)가 흔한데, 엔진이 SMILES를 **URL 경로에 그대로 박아** 넣어 경로가 깨졌다. PubChem이 받은 건 질문한 분자가 아니었다. POST 본문으로 전송해 해결.
- **모달리티 가드에 구멍**: `generate_vaccine_report`는 도킹 잡을 거부하는데, `generate_decision_report`는 **백신 잡을 받아들여** 저분자 결정 리포트로 써냈다 — 사용자가 요청한 적 없는 치료법을 조용히 보고하는 것. **한쪽으로만 작동하는 가드는 가드가 아니다.**

## Test 9 — 동시성

15개 독립 MCP 세션 동시 실행 → **예외 0, 10초 초과 0**, 중앙값 0.03초 / 최대 8.20초.

> 이 테스트가 잡은 가장 심각한 결함은 **이벤트 루프 기아**였다. AI 요약(동기 OpenAI 호출)과 리포트 생성(HTML/PDF)이 `to_thread` 없이 이벤트 루프 위에서 직접 돌아, 그 몇 초 동안 **서버 전체가 정지**했다 — 그 잡 자신의 상태 조회조차 응답하지 못했다. 카카오에서 **진행 중인 잡이 죽은 잡으로 보이던** 원인. 이벤트 루프 정지 **4,141ms → 188ms**.

## Test 10~12 — 재시도 / 복구 / 카오스

`HTTP_CHAOS_DELAY_MS` / `HTTP_CHAOS_FAIL_RATE`로 외부 API를 **실제로 느리게/죽게** 만든다(기본값 OFF, 프로덕션 동작 불변).

| 시나리오 | 결과 |
|---|---|
| 전 API 4초 지연 | 행 0건 — 일부는 정상 응답, 일부는 `isError`로 우아하게 실패 |
| 전 API 60% 강제 실패 | 행 0건 — 전부 `isError`, 외부 의존 없는 `predict_admet_profile`은 정상 동작 |

기준은 "전부 성공"이 아니라 **"매달리지 않는다"**이다. 실패는 허용된다. 10초 매달림은 카카오에서 사망이다.

**이 테스트가 잡은 실제 설계 약점:** 두 시도 모두 3초 상한이라, **꾸준히 4초 걸리는(하지만 살아있는) API를 두 번 다 타임아웃**시켜 답을 줄 수 있는 서버를 우리가 포기했다. 1차 시도는 **스톨 감지기**(3초 컷 후 재연결), 2차 시도는 **답이 실제로 와야 하는 곳**(남은 예산 전부)으로 분리.

## 메모리 누수 — 실재했고, 측정했고, 고쳤다

[`scripts/mcp_leak.py`](../backend/scripts/mcp_leak.py). 사이클마다 도킹 잡 1개 + 백신 잡 1개 + 조회 3건을 돌리고 uvicorn 프로세스의 RSS를 표본한다. 통과 조건은 "RSS가 안 오른다"가 아니라 **"워밍업 후 평탄화된다"**이다 — 인터프리터·MHCflurry 모델 로드·연결 풀은 앞쪽에서 한 번 오르고 끝나야 하고, **잡 수에 비례해 계속 오르면 그게 누수**다.

### 원인 ①: 잡 스토어에 eviction이 **전혀 없었다**

`_DRUG_DISCOVERY_STORE`는 실행된 모든 잡을 **프로세스 수명 내내** 붙들었다. 작은 누수가 아니다 — 도킹 결과 하나가 타겟의 **PDB 원문 전체**(SARS-CoV-2 스파이크 = **1.6 MB**)와 후보별 도킹 포즈를 담는다. MCP 프로세스는 카카오 쪽에서 오래 살아 있으므로, **죽을 때까지 자라기만 했다.**

수정: 오래된 것부터 eviction하되 **끝난 잡만** — RUNNING 잡은 여전히 폴링 중이므로 자기 상태 조회 밑에서 사라져선 안 된다. 상한 50. 이후 잡 수는 **50에서, 붙들린 구조 데이터는 38.3 MB에서 고정**된다.

### 그런데 그것으로 끝이 아니었다 — 원인 ②: **리포트 생성의 네이티브 메모리**

여기서 멈추고 "해결됐다"고 적었던 것이 **이 문서의 이전 판이 틀린 지점**이다. 당시엔 40사이클까지만 돌려 보고 잔여 +0.25 MB/사이클을 "할당자 단편화"라고 **단정**했다. 근거는 없었다. 그저 잡 스토어가 상한에 걸린 것을 보고 나머지도 설명됐다고 **가정**했을 뿐이다.

60사이클을 끝까지 돌리자 가정이 무너졌다:

```
총 증가: +188.1 MB / 60사이클
후반 기울기: +1.67 MB/사이클   ← 잡 수는 24사이클부터 50에 고정인데도 계속 증가
판정: ✗ 누수
```

**잡 스토어 상한은 잘 돌고 있었다. 그것이 누수의 전부가 아니었을 뿐이다.**

범인 추적 — 추측하지 않고 측정했다:

| 단계 | 방법 | 결과 |
|---|---|---|
| 어느 워크로드인가 | [`mcp_leak_attrib.py`](../backend/scripts/mcp_leak_attrib.py) — 워크로드를 **단독으로** 실행 | **도킹 잡 +0.93 MB/사이클.** 백신 잡·조회 툴·유휴 대조군은 전부 평탄 |
| 파이썬 객체인가 | `tracemalloc` | **파이썬 힙 +0.02 MB/사이클 = 사실상 0.** 아무것도 붙들고 있지 않다 → **네이티브 메모리** |
| 파이프라인 어디인가 | 단계별 격리 | RDKit·PDB 파싱·리셉터 준비 전부 평탄 |
| 리포트 생성인가 | `generate_drug_report`를 스텁으로 교체 | **+0.52 → +0.05 MB/사이클. 리포트가 누수의 ~90%** |

`reportlab`/`jinja2`가 리포트 한 건당 약 0.5 MB의 **네이티브** 메모리를 붙든다. **이건 파이썬에서 고칠 수 없다** — `gc.collect()`는 C/C++ 힙을 건드리지 않고, 해제된 네이티브 블록은 OS로 반환되지 않고 힙을 파편화시킬 뿐이다. 그 메모리를 돌려받는 유일한 방법은 **그것을 쥔 프로세스가 죽는 것**이다.

수정: [`services/report_worker.py`](../backend/services/report_worker.py) — 리포트를 **20건마다 은퇴하는 자식 프로세스**에서 렌더링한다. `max_tasks_per_child`가 핵심이다. 평범한 ProcessPoolExecutor는 **똑같은 누수를 죽지 않는 자식으로 옮길 뿐**이다.

| 60사이클 지속 부하 | 수정 전 | 수정 후 |
|---|---|---|
| 총 증가 | **+188.1 MB** | **+46.4 MB** |
| 후반 기울기 | **+1.67 MB/사이클 (평탄화 없음)** | **+0.26 MB/사이클** |
| 잡 수 | 50 고정 (상한은 정상 작동) | 50 고정 |
| 판정 | **✗ 누수** | **✓ 평탄화** |

> **fork가 아니라 spawn으로 못박았다.** 리눅스(배포 컨테이너)의 기본값은 fork인데, 이벤트 루프와 uvicorn 스레드풀이 살아 있는 프로세스를 fork하면 **다른 스레드가 쥔 락이 잠긴 채로 자식에게 복사되어 영원히 풀리지 않는다.** 자식은 첫 logging/import에서 데드락한다. 타이밍 의존이라 "가끔 리포트가 영영 안 돌아온다"로 나타난다. **이 개발 머신은 윈도우라 spawn밖에 없다 — 로컬 테스트로는 원리적으로 잡을 수 없는 버그였다.** `mp_context="spawn"`을 명시해 컨테이너가 테스트와 같은 방식으로 돌게 고정했다.

> 잔여 +0.26 MB/사이클은 여전히 0이 아니다. 1,000잡이면 약 260 MB다. 재시작 없이 무한히 도는 서버라면 언젠가 다시 봐야 한다. **"해결됐다"가 아니라 "6.4배 줄었고 아직 0은 아니다"가 정확한 서술이다** — 이 문서가 앞서 저지른 실수를 반복하지 않기 위해 적어 둔다.

### 하네스가 스스로를 속일 뻔했다 (두 번)

**첫째,** 이 스크립트는 **785 MB를 붙들고 있는 서버를 "0.4 MB, 완전 평탄"**이라고 보고했다. `tasklist`가 워킹셋을 `"804,108 K"`처럼 **따옴표 안에 천 단위 쉼표를 넣어** 출력하는데, 줄을 `,`로 쪼개면 그 숫자가 반토막 난다. CSV로 파싱해 해결.

**둘째,** 수정을 검증하는 하네스가 **1회성 풀 생성 비용(3 MB)을 사이클 수로 나눠** "+0.34 MB/사이클 — 여전히 누수"라고 보고했다. 실제 데이터는 1사이클 147.7 MB → 12사이클 147.6 MB로 **완전히 평탄**했다. **1회성 비용은 누수가 아니다.** 정상 상태 기울기로 고쳐 측정.

**조용히 틀린 숫자를 읽는 측정 도구는 없느니만 못하다.** 두 번 다 판정을 뒤집을 뻔했다.

## 해당 없음 (그리고 그 이유)

정직하게 적는다. 없는 컴포넌트를 테스트했다고 보고하지 않는다.

| 제안된 테스트 | 왜 해당 없나 |
|---|---|
| Redis Flush / Browser Cache 삭제 | Redis를 쓰지 않는다. 캐시는 전부 프로세스 메모리 → **재시작이 완전한 flush**. |
| Test 6/7/15 — Context Memory, Multi-turn, Long Context | **MCP 서버는 대화 상태를 갖지 않는다.** 히스토리는 카카오(호스트)가 관리한다. 서버가 기억하는 건 `job_id`로 키가 잡힌 잡 스토어뿐이고, 그것은 Test 13(격리)과 Test 8(체이닝)으로 검증된다. 서버가 갖지 않은 컨텍스트를 "유지하는지" 시험할 수는 없다. |
| Test 14 — Idempotency | 잡 시작 툴은 **의도적으로 비멱등**이다(`idempotentHint=false`). 같은 요청은 새 `job_id`를 만든다 — 백그라운드 잡의 올바른 계약이다. **결과의 동일성**이 실제로 중요한 것이고, Test 18에서 검증했다. |
| Test 16 — Registry (등록/삭제/Disable/Version) | 툴은 `@mcp.tool` 데코레이터로 **코드에 정적으로 정의**된다. 런타임 레지스트리가 없으므로 런타임 등록/삭제도 없다. |
| Test 11 — 특정 MCP Disable | 개별 툴을 끄는 스위치가 없다. 대신 **의미 있는 등가물**인 "그 툴이 의존하는 외부 API를 죽인다"를 Test 12(카오스)로 수행했다 — 실제 장애 모드에 훨씬 가깝다. |
| Test 20 — Logging & Audit | 로깅은 존재하나(툴명/지연/에러/job_id) 감사 요구사항이 정의된 바 없어, **통과/실패를 판정할 기준이 없다.** 기준 없이 "통과"라고 쓰지 않는다. |

## 미해결 / 알려진 한계

- **AutoDock Vina 바이너리가 어디에도 없다 — 로컬에도, 배포본에도.** `bin/`이 없고 Dockerfile도 설치하지 않으며 `requirements.txt`는 의도적으로 제외한다고 명시한다. 따라서 **모든 도킹 점수는 결정론적 휴리스틱**이며, `source` 필드에 정직하게 표시된다. **의도된 유예**다(Vina는 나중에 추가 예정 — 도킹 실행 시간이 길다). 이 문서의 이전 판은 "6개 타겟은 실제 Vina 캐시값"이라고 적었으나 그 구분을 뒷받침하는 코드 경로는 확인되지 않았다.

  **그 결과로 생기는, 놓치기 쉬운 사실:** 저분자 파이프라인 1/6단계는 **실제로 AlphaFold DB 구조를 가져오고**(항목이 없으면 UniProt 서열을 받아 ESMFold로 예측 — 실제 사례: 인플루엔자 M2 `P06821`), residue 수를 세고, meeko로 **진짜 리셉터와 블라인드 도킹 박스까지 만든다.** 그런데 Vina가 없으면 `docking_engine._heuristic_binding_score(mol_wt, log_p, rot_bonds)`로 떨어지는데 — **인자에 타겟 구조가 없다.** 즉 **affinity 숫자는 리간드 성질만으로 계산되며, 애써 가져온 구조는 점수에 전혀 들어가지 않는다.** 포즈가 없으니 포즈 기반 결합 포켓 분석도 null이다.

  그러므로 현재 정확한 표현은 **"AlphaFold 구조 확보 · 리셉터 준비까지"**이고, **"AlphaFold 기반 도킹 점수"는 Vina를 넣은 뒤에야 참이 된다.** 2024 노벨화학상 기술을 근거로 대외 설명을 할 때 이 구분을 지켜야 한다. Vina를 추가하면 구조 → 실제 도킹 물리 → 실제 포즈 → 포켓 분석으로 고리가 닫힌다.

  (백신/신항원 트랙은 AlphaFold를 아예 쓰지 않는다 — 펩타이드·MHC 수준에서 끝난다. 이 한계는 저분자 트랙에만 해당한다.)
- **Docker 이미지가 한 번도 빌드/부팅된 적이 없다.** 이 머신에는 Docker도 WSL도 없다. [`deploy_preflight.py`](../backend/scripts/deploy_preflight.py)가 정적으로 잡을 수 있는 것(윈도우 전용 경로, PATH 해석, 이미지에 담기는 파일, 툴 설명 길이)은 전부 통과하지만, **정적 검사는 "코드가 PATH를 뒤진다"는 것만 증명할 뿐 "그 PATH에 바이너리가 있다"는 것은 증명하지 못한다.** 빌드 중 `mhcflurry-downloads fetch`가 실제로 성공하는지, 리눅스에서 meeko/RDKit이 올라오는지는 미검증. **남은 가장 큰 리스크다** — 이 세션에서 찾은 가장 심각한 버그 두 개(윈도우 전용 리셉터 경로, fork 데드락)가 정확히 이 계층에 있었다.
- 잡 스토어는 **50개**로 제한된다. 51번째 잡을 시작하면 가장 오래된 **끝난** 잡이 사라지므로, 그 `job_id`로 리포트를 다시 생성할 수 없다. 카카오의 대화 수명을 감안하면 넉넉하지만, 무제한은 아니다.
- 잔여 메모리 드리프트 **+0.26 MB/사이클** — 6.4배 줄었으나 0은 아니다. 1,000잡 ≈ 260 MB.
- 카카오의 실제 **응답 크기 상한은 비공개**다. `--max-chars`(기본 4,000)는 가정치이며, 그보다 낮다면 다시 잘릴 수 있다. 그래서 모든 리포트는 **결론을 맨 앞에** 둔다 — 잘림은 항상 뒤에서 일어나므로, 뒤에 놓인 결론은 사라지는 결론이다.
